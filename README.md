## 1. Claude Design とは（まず全体像）

Claude Design は、ひとことで言うと **「言葉で頼むとデザインを作ってくれる Claude」** です。Anthropic の実験的プロダクト群「Anthropic Labs」から出ているサービスで、チャットのように「こういう LP を作って」「この資料をスライドにして」とお願いすると、Claude が下書きを作り、そこから直してもらったり、書き出したりできます ✅

料理にたとえると、これまでのデザイン作業が「自分で包丁を握って一品ずつ作る」だったのに対し、Claude Design は **「材料と“こういう雰囲気で”を伝えると、たたき台が出てくる」** 感覚に近いです。出てきたものを見ながら「ここの色を落ち着かせて」「文字を大きく」と会話で整えていきます。専門的なデザインソフトの操作を覚える必要はありません。

ポイントは、Claude Design が単なる「画像ジェネレーター」ではないことです。中身は **「プロジェクト」と「デザインシステム」という2つの層** で考えられています 🔶。

- **デザインシステム** … よく使う色・フォント（タイポグラフィ）・部品（ボタンやカードなどのコンポーネント）を“ルール集”としてまとめたもの。たとえば「自社のブランドの世界観」「営業資料のトンマナ」などを、ここに定義しておきます。
- **プロジェクト** … そのデザインシステムを反映して作る実際の成果物。LP、アプリ画面、スライド、ワイヤーフレームなど。

先に「自分のルール（デザインシステム）」を用意しておくと、あとはプロジェクトをいくつ作っても、色・フォント・部品が**自動でそろう**——これが Claude Design の設計思想です ✅。

**誰向けか**：デザイナーだけでなく、マーケター・プロダクトマネージャー・エンジニア・個人事業主まで。「デザインの専門家ではないが、見栄えのいいものを早く作りたい人」に向いています ✅。

## 2. 何ができるツールなのか（入力と出力）

### 作れるもの（出力）✅

- **プロトタイプ／インタラクティブなモックアップ**（静的な画面を、実際に触れる試作品に）
- **ワイヤーフレーム・UIモックアップ**（アプリやWebの画面設計）
- **スライド・ピッチデック**（プレゼン資料）
- **1枚資料・ドキュメント**（職務経歴書、サマリー、マーケ資料）
- **デザインの“振り幅”出し**（同じ依頼から複数方向のデザイン案）
- **アニメーション**やコードを使った表現（音声・動画・3D などを含む試作）

### 渡せるもの（入力）✅

- **テキストの指示**（「こういう LP を」）
- **画像・文書のアップロード**（DOCX / PPTX / XLSX など）
- **Webサイトのキャプチャ取り込み**（既存サイトの要素を取り込む）
- **コードベース／デザインファイル**（後述のデザインシステム取り込み）

### 出せる先（書き出し）✅

PDF・PowerPoint・HTML での書き出しに加え、外部ツールへの受け渡しに対応します。2026年6月のアップデートで連携先が拡大し、**Adobe / Base44 / Canva / Gamma / Lovable / Miro / Replit / Vercel / Wix** に渡せるようになりました（くわしくは §3）。

> 公式は「他のツールでは20以上のプロンプトが必要だったページが、Claude Design では2プロンプトで済んだ」という利用者の声を紹介しています ✅（あくまで一例で、すべてのケースに当てはまるわけではありません）。

## 3. 2026年6月の大型アップデート：4つの進化

このアップデートは、Claude Design を「目新しいジェネレーター」から「日々の仕事に組み込めるワークフロー製品」へ押し上げる内容でした 🔶。アプリ内のお知らせ（What's new in Claude Design）でも、次の4点が新機能として案内されています ✅。

### 進化① 自分のデザインシステムを取り込める（Build in your design system）✅

**GitHub リポジトリ・デザインファイル・ローカルのコードベース**から、色・タイポグラフィ・コンポーネントを読み取って取り込めるようになりました。取り込んだあとは、Claude が**“自社の本物の部品”を使って**デザインを作り、生成結果を自分のデザインシステムと照らし合わせて、ズレていれば自動で直してから見せてくれます。

開発メディアの検証では、サイトのリポジトリを指定すると **約90秒**で実プロダクトに合ったスタイルガイドが生成され、`tokens.json`（色・文字サイズ・余白・角丸）・`components/` フォルダ・`guidelines.md`（デザインの方針）といった成果物が出てきたと報告されています 🔶。Tailwind の設定やコンポーネント群からトークンを抽出するイメージです。

大きな組織向けには、**管理者が「標準のデザインシステム」を1つ承認して固定（ロック）**できる仕組みも入りました。これで、誰が作っても会社のガイドラインに沿った成果物になります ✅。

### 進化② canvas 上で直接編集できる（Edit directly on the canvas）✅

これまでは「プロンプトで直す」が基本でしたが、新しいレイアウトコントロールで、**要素をドラッグ・リサイズ・整列**と、見たまま（WYSIWYG）で直せるようになりました。「あと少し右」「ここだけ大きく」のような微調整を、いちいち言葉にしなくてもよくなったのが効きます。Anthropic のプロダクトマネージャーが新しい編集体験を“fire（最高）”と評したと報じられています 🔶。

### 進化③ Claude Code と双方向に同期する（後述 §4 で詳しく）✅

Claude Design と Claude Code が**両方向で同期**できるようになりました。`/design-sync` でデザインシステムをリポジトリ／Claude Design に引き込み、`/design` でターミナルからデザインの作成・編集・同期ができます。デザイン→実装が一本の線でつながります。

### 進化④ 使用上限の改善と、書き出し先の拡大 ✅

- **使用上限**：これまで Claude Design は単体で枠を消費し、いわゆる“トークンを食いすぎる”問題が指摘されていました。アップデートで **Claude 全体（チャットや Claude Code など）と使用上限の枠（usage pool）を共有**するようになり、作業の余地が増えました ✅。
- **書き出し先**：PDF・PowerPoint に加え、**Adobe・Base44・Canva・Gamma・Lovable・Miro・Replit・Vercel・Wix** へハンドオフできるようになりました。作ったあと「普段使っているツールに持っていく」が現実的になっています ✅。

## 4. Claude Code との連携（ここが本命）

AIエージェントや Claude Code で開発・自動化を進めるチーム・組織にとって、今回いちばん効くのがここです。**Claude Design に取り込んだデザインシステムは、Claude Code が実装に使うコンポーネントライブラリと同じもの**になります ✅。つまり、デザインとコードが別々の世界ではなく、同じ部品を共有する1つの世界になります。

### 使うコマンドは2つ ✅

- **`/design-sync`** … デザインシステムをリポジトリへ／Claude Design へ引き込む（同期）。自分が作ったものを Claude Design に押し戻すこともできる。
- **`/design`** … ターミナル（Claude Code）から、デザインプロジェクトを作成・編集・同期する。

> ※ ローカルのコードや GitHub・各種コネクタとの連携は、**Claude のデスクトップアプリ**から行います（ブラウザだけだとローカルのファイルにアクセスできません）。この資料のプロンプトは、デスクトップアプリの Claude Design に貼って使う前提です。

### 往復（ラウンドトリップ）の流れ 🔶

1. Claude Design で **デザインシステム**を整える（または既存リポジトリから取り込む）
2. その上で **プロジェクト**（画面・スライド・LP など）を視覚的に作り込む
3. できあがりを **ハンドオフ用バンドル**にまとめ、Claude Code に「これを実装して」と1つの指示で渡す
4. 実装側の変更も `/design-sync` で Claude Design に戻せるので、**デザインとコードがズレない**

これまでは「デザインを画像で書き出す → エンジニアが目で見て実装し直す」という手渡しが普通でした。今回の連携は、その手渡しの“コピペ”をなくし、**デザインとコードを地続き**にするものです。Claude Code を日常的に使っている人ほど、恩恵が大きい構造になっています。

> 動画の解説（まさおAIじっくり解説ch・2026/06/19）でも、「デザインシステムを作り込んだ上でプロジェクトに展開し、それを Claude Code に渡して視覚的にレビューしながら進める」という流れが、今回のアップデートの肝として紹介されています 🔶。

## 5. “改めて” Claude Design の良さ — なぜ今これが効くのか

最新アップデートを踏まえて、Claude Design の価値を3つに整理します。

### 良さ① 「AIっぽい量産デザイン」から抜け出せる 🔶

AI でデザインを作ると、どうしても**“どれも似たような見た目”**になりがちで、他と差がつきません。Claude Design は、**自分のデザインシステム（自社の色・フォント・部品）を土台にする**ことで、出てくるデザインを“自分たちの顔”に寄せられます。実務者のあいだでは、画像生成ツールでトンマナを効かせた素材を先に作り、それを読み込ませて“AIっぽさ”を抜く、という合わせ技も使われています（§8）。

### 良さ② 「作って終わり」ではなく「実装まで地続き」 ✅

多くのデザインツールは、きれいな画像は出せても、そこから先（コード化）が分断していました。Claude Design は Claude Code と同じ部品を共有し、ハンドオフバンドルで実装まで一本でつながる。**“見た目”と“動くもの”の距離が近い**のが、エンジニア視点での本当の強みです。

### 良さ③ 「ブランドが崩れない」仕組みがある ✅

個人なら自分のルール、チーム/企業なら管理者が承認した標準デザインシステムを固定できるので、**誰が作ってもトンマナが揃う**。「AIに任せると統一感がなくなる」という不安に、仕組みで答えているのが今回の方向性です（公式ブログのタイトルも「Claude Design が日々の仕事でブランドを守る」という主旨）🔶。

## 7. 使い方（初心者基準）

むずかしく考えなくて大丈夫です。**「あなたは“監督”、Claude が“制作スタッフ”」**くらいの感覚で、やりたいことを伝えるだけです。

### まず前提：Claude のデスクトップアプリで使う

Claude Design は **Claude のデスクトップアプリ**（Mac / Windows）で開くのがおすすめです。デスクトップアプリなら、**ローカルのコード・GitHub・各種コネクタ**とつなげられるので、後述の「自社のデザインシステムを取り込む」が活きます。この資料のプロンプト例は、すべて**デスクトップアプリの Claude Design に貼って使う**前提で書いています。

### おすすめの順番：いきなりデザインを作らない

Claude Design は、**① デザインシステム → ② ワイヤーフレーム → ③ デザイン**の順で作るのが、結局いちばん早く・崩れません。最初に「色・部品の土台（デザインシステム）」を用意し、次に「画面の骨組み（ワイヤーフレーム）」で構成を固めてから、デザインに起こします。最後に **Claude Code へハンドオフ**して実装します。いきなり完成形のデザインから入ると、後でブランドがズレたり作り直しになりがちです。

### こんな悩み → こう頼むだけ（対応表）

| こんなとき | こう頼む（例・デスクトップアプリの Claude Design に貼る） |
|---|---|
| いきなり作ると崩れる | 「先に色・部品のデザインシステムを作って。次にワイヤーフレームで構成を固めてから、デザインにして」 |
| 毎回デザインがバラつく | 「うちの GitHub リポジトリ（またはローカルのコード）を読み込んで、デザインシステムを作って。次からこれを使って」 |
| 画面の骨組みから作りたい | 「{業種}向け LP のワイヤーフレームを、今のデザインシステムに沿って作って。構成は FV→課題→特徴3つ→事例→申し込み」 |
| 細かい位置を直したい | （canvas で）要素をドラッグ・リサイズして整える／「この見出しだけ大きく」 |
| そのまま実装に渡したい | 「これをハンドオフ用にまとめて」→ デスクトップアプリから Claude Code で `/design` を使って受け取る |

### そのままコピペで試せるプロンプト例

すべて **Claude のデスクトップアプリの Claude Design** に貼って使う前提です。①→②→③ の順で送ります。

```text
# ① まずデザインシステムを作る
このGitHubリポジトリ（またはローカルのコード）を読み込んで、
色・タイポグラフィ・コンポーネントのデザインシステムを作って。
```

```text
# ② ワイヤーフレーム（骨組み）を作る
{自分のサービス}のLPのワイヤーフレームを、今のデザインシステムに沿って作って。
構成は「ファーストビュー → 課題 → 特徴3つ → 事例 → 申し込み」。配色・余白はシステム準拠で。
```

```text
# ③ デザインに仕上げる → ④ ハンドオフ（実装は Claude Code へ）
このワイヤーフレームをデザインに仕上げて。完成したらハンドオフ用にまとめて。
（書き出したいときは「PowerPoint / PDF に書き出して」）
```

### 最初の4ステップ

1. Claude の**デスクトップアプリ**で Claude Design を開く（ローカル / GitHub 連携が使える）
2. まず**デザインシステム**を作る（または既存リポジトリ・コードベースから取り込む）
3. **ワイヤーフレーム**で画面の構成を固めてから、デザインに起こす
4. canvas で整え、デスクトップアプリから **Claude Code へハンドオフ**して実装する

> 重い操作を覚える必要はありません。まずは「デザインシステム → ワイヤーフレーム」の土台づくりから始めるのが、つまずかないコツです。

## 8. 現場の知見（実務者・動画の声）

参考にした動画・実務者の声から、**複数で共通していたポイント**だけを抜き出します（個別の断定や数値は確度を下げて扱っています）。

- **二層構造（デザインシステム ↔ プロジェクト）を最初に理解すると、一気に分かりやすくなる** 🔶。先にルールを作り、その上にプロジェクトを積む、という順番。
- **Claude Design 単体だとデザインが似通いがち**。画像生成ツールでトンマナを効かせた素材を先に作り、それを読み込ませると“AIっぽさ”が抜けて差別化しやすい 🔶（あるクリエイターは、この手法で外注なら高額になるWebサイトを数時間で作ったと紹介）。
- **最初は複雑に感じる**。プロジェクトとデザインシステムの関係に慣れるまで、少し戸惑う人が多い 🔶。
- **“ベストプラクティスは自分で育てる”**。新機能が次々出るので、固定のやり方を待つより、自分の業務に合わせて型（スキル）を作り、結果を見て直していくのが結局いちばん早い 🔶。

> ⚠️ これらは動画・SNS など二次情報を含みます。固有の数値（〇時間で完成・〇割短縮など）は出典が弱いものは本文で断定していません。実際の効果は使い方しだいです。

## 9. ユースケース集

- **LP・Webサイト** … たたき台を出して会話で整え、画像素材でトンマナを足し、最後に Vercel などへ書き出す
- **スライド・営業資料** … 構成を伝えて生成 → canvas で微調整 → PowerPoint / PDF へ
- **プロトタイプ・アプリUI** … ワイヤーフレーム → 触れる試作品 → ハンドオフして Claude Code で実装
- **デザインシステム構築** … 既存リポジトリを取り込み、`tokens.json` 等を生成して全社の土台に
- **資料・1枚もの** … 職務経歴書、サマリー、インフォグラフィックなどを素早く

## 10. もう一歩進んだ使い方（開発者向け）

ここは Claude Code を使う人向けの発展編です（本筋は §1〜9 で十分です）。

- **デザインを“コードのように”扱う**：取り込み時に生成される `tokens.json` / `components/` / `guidelines.md` を、リポジトリ管理下に置けば、デザインも差分（diff）で追える素材になります 🔶。
- **往復を習慣化する**：実装で部品を更新したら `/design-sync` で Claude Design 側へ戻し、デザインとコードの“真実の источник（source of truth）”を1つに保つ ✅。
- **ハンドオフは1指示で**：デザインをバンドルにまとめ、Claude Code に「このバンドルを実装して」と渡す。レビューは視覚的に行い、ズレたら canvas で直して再同期 🔶。
- **チームで標準化**：管理者が標準デザインシステムを承認・固定 → 誰が作っても崩れない土台にする ✅。

## 12. FAQ

**Q. 初心者でも使えますか？**
A. はい。デザインソフトの操作は不要で、言葉で頼んで会話で整えるだけです。まずは1枚資料から始めるのがおすすめです。

**Q. 無料で使えますか？**
A. Claude の Pro / Max / Team / Enterprise プランで利用できます（ベータ）。Enterprise は管理者が有効化する必要があります。最新の提供条件は公式で確認してください。

**Q. Figma の代わりになりますか？**
A. 役割が一部重なりますが、Claude Design は「会話で作って、コードまで地続きにする」点が特徴です。Figma などとは書き出し・連携で組み合わせることもできます。断定的な優劣はここでは付けません。

**Q. デザインが毎回似てしまいます。**
A. デザインシステムで自分の色・部品を土台にする、画像生成でトンマナ素材を先に用意して読み込ませる、などで差別化できます（§8）。

**Q. Claude Code がないと使えませんか？**
A. いいえ。Claude Design 単体でも使えます。Claude Code との連携は“実装まで地続きにしたい人”向けの強みです。
